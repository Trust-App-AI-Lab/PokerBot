Data Flow (Two-Port Model)

CoachBot/CC only touches :3456 (relay). BotManager only touches :3457 (server). Never cross.

DataSourceConsumerEndpoint
User game state relay CoachBot / CC GET :3456/state (no player field needed)
Bot game state server BotManager GET :3457/state?player=X
User actions CoachBot relay → server POST :3456/action (no player field)
Bot actions PlayBot server POST :3457/action (player field required)
GTO analysis /poker-strategy tools CoachBot / PlayBot python3 <SKILL_DIR>/tools/*.py
Hand history relay (isolated) Review mode GET :3456/history?last=N
Game config host server GET/POST :3457/config
Join / Start / Rebuy CC (setup) server POST :3457/join /start /rebuy

Three Roles

CoachBot
Main session, real-time GTO coach
Observer only (manual) / Proxy (auto)
Reads :3456/state
Every msg: 🃏 CoachBot:
Bilingual (auto-detect)
BotManager
Background process (botmanager.sh)
Stateless polling loop (2s)
claude -p --resume per turn
Reads :3457/state directly
~8-10s per bot decision
PlayBot
Individual AI personality
Persistent session (MD5 UUID)
Init: load personality + /poker-strategy tier
Turn: hot resume, decide, POST, EXIT
Information isolated

Key Invariants

1. CC always reads/writes via localhost:3456 (relay layer), regardless of backend
2. Each bot only sees its own cards (information isolation)
3. CoachBot ≠ game orchestrator — CC reads modes.md on-demand for game flow
4. Cross-skill references use /skillname, never hardcoded paths
5. bash stop-game.sh handles ALL modes, one command full cleanup
