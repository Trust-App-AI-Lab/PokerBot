Poker Now Bot Runtime Engine

WebSocket engine, dual-session architecture, browser bridge — everything that connects AI bots to pokernow.com

Game Setup Flow

0

Get Game Room

Path A: User creates room on pokernow.com → CC detects URL.
Path B: User pastes existing room link → CC extracts gameId.

↓
1

Select Bots

CC scans bot_profiles/, lists available bots. User picks which to join + whether to enable CoachBot.

↓
2

Write game.json + Launch

CC writes game config (gameUrl, bots[], coach, stack). Launches orchestrator.js (bots connect via WebSocket).

↓
3

Approve Bots + Inject Bridge

CC approves bot join requests via hostAction(). Injects coach-bridge.js into browser tab (one time). Starts coach-server.js.

↓
4

Dual-Session Game Loop

CC launches botmanager.sh & in background. Main session stays free for user chat + CoachBot coaching. BotManager handles all bot decisions autonomously.

Scripts

Core

orchestrator.js

Multi-bot WebSocket manager. Reads game.json, connects all bots (3s apart), watches for turns, writes pending-turns.json + per-bot turn.json, reads action.json, executes via WebSocket. Auto check/fold after 60s timeout.
655 lines
Core

coach-bridge.js

Browser-injected CoachBot bridge. Hooks page WebSocket, exposes window.__coach API (getState, act, hostAction, fetchLog). Pushes state to coach-server, polls /action for commands. Injected once via javascript_tool at game start.
838 lines · minified version: coach-bridge.min.js
Core

coach-server.js

HTTP server (localhost:3456) bridging CC and browser. Receives state pushes from bridge → preprocesses → writes state.json. Routes action POST from CC → bridge polls and executes. Endpoints: /state, /action, /event, /turn, /health.
360 lines · Auto-kills old instance via PID file
Background

botmanager.sh

Bash outer loop for BotManager. Polls pending-turns.json every 2s. When turns found, invokes claude -p with botmanager-prompt.md. Exits when game.json is deleted.
83 lines · Spawns stateless CC sessions

CLI interface with action validation. Validates action JSON format, checks against valid actions for current game state. Used for testing and manual action injection.

Information Isolation (3 Layers)

Layer 1: Data (Orchestrator)

Each bot's turn.json only contains that bot's own hole cards. No cross-bot data leakage at the WebSocket level.

Layer 2: Prompt (BotManager)

Subagent prompts contain NO file paths, NO directory names, NO other bot names. All data inlined as plain text. Zero filesystem knowledge.

Layer 3: Session (Dual-session)

CoachBot (main session) sees user's cards via bridge. Bot decisions run in separate claude -p sessions. User's cards never enter any bot's prompt.
